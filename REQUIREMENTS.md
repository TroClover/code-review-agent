# BRT Code Review Agent — 需求文档 v2

## 1. 项目背景

BRT（Better Release Testing）是自动驾驶公司内部负责发布测试的部门。团队日常产生大量脚本代码（Python + C++），提交至 GitHub。人工代码审查已成为研发流程中的主要瓶颈：

- **实习生**：代码量大、质量参差不齐。每个新人都要经历类似的低质量代码阶段，审查人员在基础问题上花费大量时间。
- **正式员工**：代码修改往往涉及多个文件和深层历史上下文。审查人员需要翻阅 git history 才能理解影响范围，效率低且容易遗漏。
- **审查人员**：被低级问题淹没，真正需要关注的架构和业务逻辑问题得不到足够重视。

**项目目标**：构建一个 LLM 驱动的智能代码审查 Agent，自动化质量检查、减轻审查人员负担，并提供团队级质量洞察。

---

## 2. 核心价值主张

### LLM 审查 vs 现有工具的定位

| 工具类型 | 擅长 | 不擅长 |
|----------|------|--------|
| **Linter**（ruff/flake8/clang-tidy） | 命名规范、格式、import 顺序、简单模式匹配 | 语义理解、业务逻辑、跨文件影响 |
| **静态分析**（SonarQube/CodeClimate） | 代码复杂度、重复代码、技术债务量化 | 上下文感知、PR 级审查、业务逻辑 |
| **人工审查** | 架构设计、业务逻辑、团队知识传承 | 一致性、速度、覆盖面 |
| **LLM 审查（本项目）** | 语义理解、逻辑正确性、上下文感知、跨文件分析 | 格式规范（交给 linter）、大规模代码库分析 |

**核心定位**：LLM 审查不是替代 linter 或人工审查，而是填补两者之间的空白——**语义级别的代码理解**。Linter 能发现的问题交给 linter，LLM 专注于 linter 无法覆盖的逻辑、安全、性能问题。

## 3. 用户角色

| 角色 | 描述 | 核心需求 |
|------|------|----------|
| 开发者 | 提交 PR 的工程师（含实习生和正式员工） | 可操作的审查意见：指出问题的同时给出具体修改建议 |
| 审查者 | 被指派审核 PR 的资深工程师 | 预过滤问题：只看到需要人工判断的高价值问题，不被琐碎问题干扰 |
| 管理员 | 负责 Agent 的配置和部署 | 易于配置、规则管理、成本监控 |

**注意**：不按人的角色（实习生/正式员工/资深工程师）区分审查策略。审查严格程度由仓库/分支的审查 profile 决定，确保审查公平性。

## 4. 功能需求

### 4.1 审查能力

| 编号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| F-RC-01 | 逻辑正确性审查：边界条件、循环终止、递归深度、异常处理 | P0 | LLM 核心价值，linter 无法覆盖 |
| F-RC-02 | 安全漏洞检测：硬编码密钥、SQL 注入、路径遍历、不安全的反序列化 | P0 | 通用安全检查，默认启用 |
| F-RC-03 | 性能问题识别：不必要的拷贝、低效算法、内存分配模式 | P1 | LLM 语义分析优势 |
| F-RC-04 | 跨文件影响分析：修改 A 文件对 B 文件的影响 | P1 | 需要理解 import/调用关系 |
| F-RC-05 | 自定义规则支持：通过 YAML 配置定义自定义审查规则 | P2 | 团队特定关注点 |
| F-RC-06 | 自动驾驶领域安全检查（可选）：传感器数据验证、仿真配置检查、安全关键路径审查 | P1 | 通过 `.breview.yml` 配置启用，默认关闭 |

### 4.2 审查范围

| 编号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| F-RS-01 | 基于 diff 的审查：仅审查变更行及其周围上下文 | P0 | 核心模式，成本最优 |
| F-RS-02 | 全文件审查：仅对新增文件审查完整内容 | P2 | 默认关闭，按需启用 |
| F-RS-03 | PR 级别审查为主，支持按逻辑变更块拆分审查 | P0 | 大 PR 中多个独立变更可分别审查 |
| F-RS-04 | 大 PR 处理：将大 PR 拆分为可管理的块，不超出 LLM token 限制 | P1 | 按文件分组，每组独立审查 |

### 4.3 上下文理解

| 编号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| F-CU-01 | 解析 git diff，提取变更文件、变更行、变更类型（新增/修改/删除） | P0 | 基础能力 |
| F-CU-02 | 包含周围代码上下文：函数体、类定义、import 语句 | P0 | 帮助 LLM 理解变更的语义 |
| F-CU-03 | 理解跨文件依赖：import、函数调用、类型引用 | P1 | 用于跨文件影响分析 |
| F-CU-04 | Token 截断策略：在 token 限制内优先保留 diff 周围代码 | P0 | 简单截断，不过度设计 |

| 编号 | 需求 | 优先级 | 说明 |
|------|------|--------|------|
| F-PR-01 | 按仓库/分支配置审查 profile（strict/standard/relaxed） | P0 | 替代按人区分角色 |
| F-PR-02 | strict profile：启用所有检查，低严重等级阈值 | P0 | 用于 main/release 分支 |
| F-PR-03 | standard profile：启用核心检查，中等阈值 | P0 | 用于 feature 分支（默认） |
| F-PR-04 | relaxed profile：仅启用高优先级检查 | P1 | 用于实验性/WIP 分支 |
| F-PR-05 | 支持在 .breview.yml 中自定义 profile 的检查项和阈值 | P1 | 团队可定制 |

## 5. 多 Agent 架构

### 5.1 架构概览

```
PR 触发（webhook / CLI）
  │
  ▼
Orchestrator Agent
  ├── 解析 PR diff
  ├── 构建上下文（含原 Context Agent 逻辑）
  ├── 运行 linter 集成
  └── 决定调用哪些 Agent
  │
  ▼ （并行）
  ├──→ Code Review Agent ─┐  两个 Agent 并行执行
  └→ Safety Agent ────────┘  各自产出 Issue 列表
         │
         ▼
Orchestrator Agent（聚合结果）
  ├── 合并 linter 结果
  ├── 去重（同一问题多个来源重复报告）
  ├── 排序（严重等级 + 相关性）
  ├── 过滤（按 profile 阈值过滤低优先级问题）
  └── 生成报告 + 发布 PR comment
```

### 5.2 Agent 清单

| Agent | 职责 | 调用策略 | LLM 策略 |
|-------|------|----------|----------|
| **Orchestrator Agent** | 总调度：解析 diff、构建上下文、运行 linter、决定调用哪些 Agent、聚合结果、生成报告 | 每次必调 | 轻量模型 / 规则 |
| **Code Review Agent** | 核心审查：逻辑正确性 + 安全漏洞 + 性能问题 | 每次必调 | 强 LLM |
| **Safety Agent** | 安全关键代码：通用安全检查 + 可选的自动驾驶领域规则 | 每次必调 | 强 LLM + 领域规则 |

### 5.3 工具层（非 Agent）

| 工具 | 职责 | 说明 |
|------|------|------|
| **Linter 集成** | 运行外部 linter（ruff/flake8/clang-tidy） | 替代原 Style Agent |
| **Diff 解析器** | 解析 git diff | 已有实现 |
| **上下文构建器** | 构建代码上下文（函数体、类定义、import） | 合并到 Orchestrator |
| **报告生成器** | 生成审查报告 | 简化为函数 |

### 5.4 协作流程

1. **触发**：PR webhook 或 CLI 命令
2. **Orchestrator**：解析 diff → 构建上下文 → 运行 linter → 选择 Agent
3. **并行审查**：Code Review Agent + Safety Agent 并行执行
4. **聚合**：Orchestrator 合并结果 → 去重 → 排序 → 过滤
5. **输出**：生成报告 → 发布 PR comment → 设置 commit status

### 5.5 失败处理

- 单个 Agent 失败不影响其他 Agent
- Orchestrator 负责容错，某个 Agent 超时或报错时跳过并标记
- LLM API 不可用时降级到 linter-only 模式

## 9. 设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 部署模式 | GitHub App + 本地 CLI | GitHub 集成用于自动 PR 审查，CLI 用于开发者提交前的本地审查 |
| 架构模式 | 3 Agent + 工具层 | 简化架构，避免过度拆分；Style Agent 被 linter 替代，Context/Report 合并到 Orchestrator |
| 审查引擎 | LLM 驱动 + linter 集成 | LLM 负责语义理解，linter 负责格式规范，各司其职 |
| 主要语言 | Python + C++ | 团队的主要开发语言 |
| 审查策略 | 按仓库/分支 profile 配置 | 确保审查公平性，避免按人贴标签 |
| 领域特化 | 通用安全检查（默认）+ 自动驾驶领域规则（可选） | 既覆盖通用场景，又保留领域差异化优势 |
| 输出语言 | 英文 | 无编码问题，代码场景通用 |
| 知识系统 | 静态知识源（CODING_STANDARD.md） | 动态知识提取是 v2+ 功能，MVP 阶段不需要 |
| 审查模式 | advisory（默认）→ 可配置为 blocking | 渐进式采纳，先建立信任 |
| 成本控制 | token 预算 + 审查缓存 | LLM 调用有真实成本，必须有控制机制 |
