# BRT Code Review Agent — 需求文档

## 1. 项目背景

BRT（Better Release Testing）是自动驾驶公司内部负责发布测试的部门。团队日常产生大量脚本代码（Python + C++），提交至 GitHub。人工代码审查已成为研发流程中的主要瓶颈：

- **实习生**：代码量大、质量参差不齐。每个新人都要经历类似的低质量代码阶段，审查人员在基础问题上花费大量时间。
- **正式员工**：代码修改往往涉及多个文件和深层历史上下文。审查人员需要翻阅 git history 才能理解影响范围，效率低且容易遗漏。
- **审查人员**：被低级问题淹没，真正需要关注的架构和业务逻辑问题得不到足够重视。

**项目目标**：构建一个 LLM 驱动的智能代码审查 Agent（多 Agent 架构），自动化质量检查、减轻审查人员负担、从审查过程中提取知识用于开发者教育，并提供团队级质量洞察。

---

## 2. 用户角色

| 角色 | 描述 | 核心需求 |
|------|------|----------|
| 实习生 | 临时贡献者，技能水平不一，提交量大 | 导师式反馈：指出问题的同时解释原因，给出修改示例 |
| 正式员工 | 有经验的开发者，代码修改常涉及历史上下文 | 上下文感知审查：理解历史变更，识别影响范围，同行级反馈 |
| 审查人员 | 被指派审核 PR 的资深工程师 | 预过滤问题：只看到需要人工判断的高价值问题，不被琐碎问题干扰 |
| 团队 Lead | 负责团队代码质量和流程管理 | 质量分析：趋势、高频问题模式、团队改进指标 |
| Agent 管理员 | 负责 Agent 的配置和部署 | 易于配置、规则管理、运行监控 |

## 3. 功能需求

### 3.1 审查能力

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-RC-01 | 代码规范检查：命名、格式、注释、import 顺序等 | P0 |
| F-RC-02 | 常见 bug 模式检测：空指针、资源泄漏、未处理异常、竞态条件 | P0 |
| F-RC-03 | 逻辑正确性审查：边界条件、循环终止、递归深度 | P0 |
| F-RC-04 | 安全性检查：硬编码密钥、SQL 注入、不安全的反序列化 | P1 |
| F-RC-05 | 性能问题识别：不必要的拷贝、低效算法、内存分配模式 | P1 |
| F-RC-06 | 跨文件影响分析：修改 A 文件对 B 文件的影响 | P1 |
| F-RC-07 | 自定义规则支持：通过 YAML 配置或 Python 插件定义自定义审查规则 | P2 |
| F-RC-08 | 安全关键代码审查：识别涉及传感器数据处理、仿真环境配置、安全验证流程的代码，应用更严格审查策略 | P1 |
| F-RC-09 | 仿真环境配置检查：仿真测试相关的配置文件、参数、环境变量的正确性检查 | P1 |

### 3.2 审查范围

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-RS-01 | 基于 diff 的审查：仅审查变更行及其周围上下文 | P0 |
| F-RS-02 | 全文件审查：按需或文件为新增时，审查完整文件 | P1 |
| F-RS-03 | PR 级别审查为主，支持按逻辑变更块拆分审查（大 PR 中多个独立逻辑变更分别审查） | P0 |
| F-RS-04 | 支持多文件 PR | P0 |
| F-RS-05 | 大 PR 处理：将大 PR 拆分为可管理的块，不超出 LLM token 限制 | P1 |

### 3.3 上下文理解

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-CU-01 | 解析 git diff，提取变更文件、变更行、变更类型（新增/修改/删除） | P0 |
| F-CU-02 | 包含周围代码上下文：函数体、类定义、import 语句 | P0 |
| F-CU-03 | 理解文件变更历史：最近的修改、贡献者、热点区域 | P1 |
| F-CU-04 | 理解跨文件依赖：import、函数调用、类型引用 | P1 |
| F-CU-05 | 解析 PR 描述和关联的 issue/ticket，理解修改意图 | P2 |
| F-CU-06 | 智能上下文窗口管理：在 token 限制内优先保留最相关的上下文 | P1 |

### 3.4 角色审查策略

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-RO-01 | 识别 PR 作者角色（实习生/正式员工/资深工程师），通过 GitHub team 或配置映射 | P0 |
| F-RO-02 | 实习生模式：逐行审查，侧重规范、基本逻辑、常见反模式；教学语气，附带解释和修改示例 | P0 |
| F-RO-03 | 正式员工模式：聚焦架构一致性、历史影响、性能、安全；同行 review 风格，直接简洁 | P0 |
| F-RO-04 | 可配置的审查深度：按角色设置检查项和跳过项 | P1 |
| F-RO-05 | 可调整的严重等级阈值：实习生更严格，资深员工更宽松 | P1 |
| F-RO-06 | 按角色选择性调用 Agent：实习生触发全部 Agent，资深员工可跳过 Style Agent 等低优先级检查 | P1 |

### 3.5 知识提取与反馈系统

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-KE-01 | 从 Agent 自身的审查历史中提取知识模式 | P0 |
| F-KE-02 | 从人工审查者的 PR comment 中提取知识，权重高于 Agent 来源 | P0 |
| F-KE-03 | 当同一问题模式出现 N 次（可配置阈值）时，自动生成知识条目 | P1 |
| F-KE-04 | 结构化存储知识：类别、严重等级、代码示例（好/坏）、适用场景、来源 | P0 |
| F-KE-05 | 两级粒度：粗粒度的团队级编码规范 + 细粒度的按问题类型/按人员的知识条目 | P0 |
| F-KE-06 | 审查时自动附带相关知识条目到 review comment 中 | P0 |
| F-KE-07 | 追踪个人知识差距：某个开发者反复违反哪些知识条目 | P1 |
| F-KE-08 | 为新入职实习生生成个性化学习清单（基于其初始代码提交） | P2 |
| F-KE-09 | 使用经用户审查通过的初始编码规范文档作为知识库 seed | P0 |
| F-KE-10 | 开发者反馈闭环：review comment 附带"有用/没用"反馈按钮，反馈数据用于优化 LLM prompt 和知识权重 | P1 |

## 4. 多 Agent 架构

### 4.1 Agent 清单

| Agent | 职责 | 调用策略 | LLM 策略 |
|-------|------|----------|----------|
| **Orchestrator Agent** | 总调度：解析 diff、识别角色、决定调用哪些 Agent、聚合结果 | 每次 PR 必调用 | 轻量模型 / 规则 |
| **Style Agent** | 代码规范：命名、格式、import、注释 | 实习生必调，正式员工可跳过 | 规则为主 + 轻量 LLM |
| **Code Review Agent** | 核心审查：逻辑正确性 + 安全漏洞 + 性能问题（3 个 sub-prompt 并行） | 每次必调 | 强 LLM |
| **Safety Agent** | 安全关键代码：传感器数据、仿真环境配置、安全验证流程、行业规范 | 每次必调 | 强 LLM + 领域规则 |
| **Context Agent** | 上下文构建：git 历史分析、代码索引、依赖关系 | 每次必调，为审查 Agent 提供输入 | 无需 LLM，纯工具 |
| **Knowledge Agent** | 知识处理：从审查结果提取知识、关联知识条目、维护知识库 | 审查完成后调用 | 轻量 LLM |
| **Report Agent** | 输出：生成报告、发布 PR comment、收集反馈、统计数据 | 审查完成后调用 | 无需 LLM |

### 4.2 协作流程

```
PR 触发（webhook / CLI）
  │
  ▼
Orchestrator Agent
  ├── 解析 PR diff
  ├── 识别作者角色
  └── 决定本轮需要调用的 Agent 列表
  │
  ▼
Context Agent（构建上下文）
  ├── git 历史分析
  ├── 代码索引查询
  └── 跨文件依赖分析
  │
  ▼ （并行）
  ├──→ Style Agent ──────┐
  ├──→ Code Review Agent ┤  三个 Agent 并行执行
  └──→ Safety Agent ─────┘  各自产出 Issue 列表
         │
         ▼
Orchestrator Agent（聚合结果）
  ├── 去重（同一问题多个 Agent 重复报告）
  ├── 排序（严重等级 + 相关性）
  └── 过滤（按角色阈值过滤低优先级问题）
  │
  ▼ （并行）
  ├──→ Knowledge Agent（知识提取与关联）
  └──→ Report Agent（PR comment 发布 + 报告生成 + 反馈收集）
```

### 4.3 Agent 间通信

- **数据格式统一**：所有 Agent 使用统一的 Issue 数据结构（定义在 Phase 0 数据模型中）
- **单向流水线**：Agent 间无复杂网状通信，只有线性流水线 + 一处并行分叉
- **失败隔离**：单个 Agent 失败不影响其他 Agent；Orchestrator 负责容错，某个 Agent 超时或报错时跳过并标记
- **并行调度**：Style / Code Review / Safety 三个 Agent 通过 asyncio 并行执行，共享 Context Agent 的输出

## 5. 非功能需求

| 编号 | 需求 | 优先级 |
|------|------|--------|
| NF-01 | 输出语言：所有审查 comment、知识条目、报告均使用英文 | P0 |
| NF-02 | 审查延迟分级 SLA：小 PR（<5 文件）≤2 分钟；中 PR（5-20 文件）≤5 分钟；大 PR（>20 文件）≤15 分钟 | P1 |
| NF-03 | Token 成本控制：可配置每次审查的 token 预算，支持缓存避免重复审查 | P1 |
| NF-04 | 数据安全：代码不得发送至不可信的第三方；支持本地 LLM 部署方案 | P1 |
| NF-05 | 可扩展性：支持新增语言、规则、LLM 后端，无需修改核心代码 | P1 |
| NF-06 | 可靠性：Agent 自身故障不应阻断 PR merge；状态检查采用 fail-open 策略 | P1 |
| NF-07 | 可伸缩性：支持多仓库并发 PR 处理 | P2 |
| NF-08 | 可观测性：日志、错误追踪、LLM 调用监控 | P1 |
| NF-09 | 审计日志：所有审查操作可追溯——谁触发、审查了什么、结果如何 | P1 |
| NF-10 | 限流与熔断：GitHub API 和 LLM API 调用限流，超限时优雅降级（排队或跳过） | P1 |
| NF-11 | 数据保留策略：审查记录和知识条目的保留周期和归档策略可配置 | P2 |
| NF-12 | 多租户隔离：多个团队使用同一实例时，数据和配置隔离 | P2 |

## 8. 已确认的设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 部署模式 | GitHub App + 本地 CLI | GitHub 集成用于自动 PR 审查，CLI 用于开发者提交前的本地审查 |
| 架构模式 | 多 Agent 协作（7 个 Agent） | 流水线 + 并行分叉，兼顾专业分工和协作简单性 |
| 审查引擎 | LLM 驱动 | 能语义理解业务逻辑和上下文，超越静态规则的局限 |
| 主要语言 | Python + C++ | 团队的主要开发语言 |
| 审查粒度 | PR 级为主 + 支持逻辑块拆分 | 大 PR 中多个独立变更可分别审查，避免上下文混杂 |
| 输出语言 | 英文 | 无编码问题，代码场景通用 |
| 知识来源 | Agent 审查 + 人工 comment（人工权重更高） | 人工反馈信噪比更高；Agent 提供覆盖面 |
| 知识粒度 | 粗粒度（团队规范）+ 细粒度（按人/按问题） | 团队一致性 + 个性化学习 |
| 知识反馈方式 | 审查时自动附带 | 无缝融入现有工作流 |
| 冷启动策略 | 用户审查通过的初始编码规范文档作为 seed | 确保质量和团队认可 |
| 安全关键代码 | Safety Agent 独立审查，仿真环境配置纳入检查范围 | 自动驾驶领域容错要求高 |
| 开发者反馈 | review comment 附带"有用/没用"按钮 | 持续优化 LLM prompt 和知识权重 |
